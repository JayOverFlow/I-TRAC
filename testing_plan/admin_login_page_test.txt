I-TRAC ADMIN LOGIN PAGE MANUAL TESTING PLAN
==========================================

Project: I-TRAC
Page: Admin Login (auth/admin-login.blade.php)
Description: Authentication page for Master Administrators

SYSTEM OVERVIEW:
- Fields: Username, Password
- Actions: Login, Link to Registration
- Validations: Required fields, credential verification, CSRF protection

---

TEST SCENARIO 1: HAPPY PATH (SUCCESSFUL LOGIN)
----------------------------------------------
Goal: Ensure a valid admin can log in successfully.
Steps:
1. Navigate to the Admin Login page.
2. Enter a valid admin Username.
3. Enter the correct Password for that account.
4. Click the "Login" button.
Expected Result: User is successfully authenticated and redirected to the Admin Dashboard.

TEST SCENARIO 2: FIELD VALIDATION (HUMAN ERRORS)
-----------------------------------------------
Goal: Ensure the system handles missing input gracefully.
- Case A (Empty Username): Leave "Username" empty, enter any password. Click "Login".
  Expected: Browser-level validation (required) or server-side error "The username field is required." appears.
- Case B (Empty Password): Enter a username, leave "Password" empty. Click "Login".
  Expected: Browser-level validation or server-side error "The password field is required." appears.
- Case C (Both Empty): Click "Login" with no fields filled.
  Expected: Form prevents submission or highlights both fields as invalid.

TEST SCENARIO 3: INVALID CREDENTIALS
------------------------------------
Goal: Test error handling for incorrect login attempts.
- Case A (Wrong Password): Enter a valid Username but an incorrect Password. Click "Login".
  Expected: Error message "These credentials do not match our records" (or similar) appears in the danger alert box.
- Case B (Non-existent Username): Enter a Username that is not registered. Click "Login".
  Expected: Error message indicating invalid credentials appears.
- Case C (Case Sensitivity): If usernames are case-sensitive, try "Admin" vs "admin".
  Expected: Consistent behavior with system security rules.

TEST SCENARIO 4: UI/UX & STICKY INPUT
-------------------------------------
Goal: Verify the user experience and data persistence.
1. Enter a valid Username but a wrong Password. Click "Login".
2. After the page reloads with an error:
   Expected: The "Username" field should still contain the value you typed (old input). The "Password" field should be empty for security.
3. Check the layout:
   Expected: Title "Master Admin Login" is visible, login button "Login" is clearly styled in red, and the I-TRAC logo is correctly displayed.

TEST SCENARIO 5: NAVIGATION
---------------------------
Goal: Ensure links to other pages work correctly.
1. Locate the text "Don't have an account? Register."
2. Click the "Register" link.
Expected Result: User is redirected to the Admin Registration page.

TEST SCENARIO 6: SECURITY & INTERRUPTIONS
-----------------------------------------
Goal: Test CSRF and session persistence.
1. Leave the login page open for an extended period (e.g., 2 hours).
2. Attempt to login.
   Expected: If the CSRF token has expired, the system should ideally redirect back to login with an error or handle it gracefully.
3. Multiple Submissions: Click "Login" multiple times rapidly.
   Expected: Only one request should be processed (ideally, the button should disable after click).
