I-TRAC REGISTRATION PAGE MANUAL TESTING PLAN
===========================================

Project: I-TRAC
Page: Registration (auth/register.blade.php)
Description: 4-Step Stepper Registration Process

SYSTEM OVERVIEW:
- Step 1: Personal Information (First Name, Middle Name, Last Name, Suffix, TUP ID)
- Step 2: Account Details (Email, Password, Confirm Password, User Type, Department)
- Step 3: Review Information
- Step 4: Email Verification (OTP)

---

TEST SCENARIO 1: HAPPY PATH (SUCCESSFUL REGISTRATION)
-----------------------------------------------------
Goal: Ensure a user can register successfully with valid inputs.
Steps:
1. Navigate to the Register page.
2. Step 1: Enter valid First, Middle, and Last name. Enter a 6-digit TUP ID. Click "Next".
3. Step 2: Enter a valid TUP email (@tup.edu.ph), matching passwords (8+ chars, 1 letter, 1 number), select a User Type and Department. Click "Next".
4. Step 3: Verify all displayed information is correct. Click "Next".
5. Modal: Click "Send Code".
6. Step 4: Enter the 6-digit code received in email.
7. Click "VERIFY".
Expected Result: User is redirected to the dashboard or success page.

TEST SCENARIO 2: STEP 1 VALIDATION & HUMAN ERRORS
-------------------------------------------------
Goal: Test error handling for personal information.
- Case A (Missing Required): Leave First Name or Last Name empty. Click "Next".
  Expected: Error message "Please enter your [field]" appears; cannot proceed.
- Case B (Invalid TUP ID): Enter "123" or "1234567" (not 6 digits).
  Expected: Field should flag as invalid (if enforced).
- Case C (Suffix Optional): Leave Suffix empty.
  Expected: Registration proceeds without error.

TEST SCENARIO 3: STEP 2 VALIDATION (ACCOUNT SECURITY)
-----------------------------------------------------
Goal: Test email and password security logic.
- Case A (Invalid Email Format): Enter "user@gmail.com" or "user@tup".
  Expected: Validation error (if @tup.edu.ph is strictly enforced).
- Case B (Password Complexity): Enter "12345678" (no letters) or "abcdefgh" (no numbers).
  Expected: Validation error indicating complexity requirements.
- Case C (Password Mismatch): Enter "Password123" and "Password321".
  Expected: Error "Passwords do not match" or similar.
- Case D (Empty Radio/Select): Proceed without selecting User Type or Department.
  Expected: "Please select a user type" and "Select Department/Office" errors.

TEST SCENARIO 4: NAVIGATION & EDITING (USER FLOW)
-------------------------------------------------
Goal: Ensure users can go back and fix mistakes.
1. Fill Step 1 and proceed to Step 2.
2. Click "Back" button.
   Expected: Returned to Step 1; previously entered data should still be there.
3. Edit Last Name and proceed to Step 3.
   Expected: Step 3 shows the updated Last Name.

TEST SCENARIO 5: REVIEW STEP FUNCTIONALITY
------------------------------------------
Goal: Verify the "Review" page behaves as expected.
1. Check if all fields match what was typed in Step 1 and 2.
2. Click the "Eye" icon next to Password.
   Expected: Password toggles between masked (dots) and plain text.
3. Verify "Next" triggers the "Send Verification Code" modal.

TEST SCENARIO 6: EMAIL VERIFICATION (OTP) ERRORS
------------------------------------------------
Goal: Test the OTP entry and resend logic.
- Case A (Incorrect Code): Enter "000000". Click "VERIFY".
  Expected: Error message "Invalid verification code".
- Case B (Incomplete Code): Fill only 3 out of 6 boxes.
  Expected: "VERIFY" button disabled or shows error.
- Case C (Resend Logic): Wait for the timer to expire. Click "Resend".
  Expected: Timer resets; new code is sent; old code might be invalidated.

TEST SCENARIO 7: FORM RESUBMISSION & INTERRUPTIONS
--------------------------------------------------
Goal: Test handling of common accidental user actions.
1. Double-clicking "VERIFY": Click the button twice rapidly.
   Expected: System should disable button after first click to prevent double submission.
2. Page Refresh: Refresh the page while on Step 4.
   Expected: User might need to restart or session might persist (check current implementation).
3. CSRF Protection: Leave the page open for 2 hours, then try to submit.
   Expected: Should handle session expiry gracefully (usually redirects to login or error).
