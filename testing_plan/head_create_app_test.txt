================================================================================
TESTING PLAN: Head - Create Annual Procurement Plan (APP) Page
================================================================================
Page Under Test: resources/views/head/pages/head-create-app.blade.php
Location: /head/create-app (or relevant route)

Objective: To ensure the functional correctness, user interface responsiveness, 
and error handling of the Create APP page for the Head role.

--------------------------------------------------------------------------------
1. UI Elements & Layout Verification
--------------------------------------------------------------------------------
[ ] Verify that the page title is displayed correctly ("Create Annual Procurement Plan | I-TRAC").
[ ] Verify that the "Allocated Budget" is visible and formatted properly (e.g., PHP 12,345.00).
[ ] Verify the presence of the "Done" and "Save as Draft" buttons at the top right.
[ ] Verify the default Project Item form (Item 1) is visible with all its fields.
[ ] Verify the "Add Project" button is visible and active at the bottom.
[ ] Verify the "Total Amount" display shows 0.00 by default at the bottom right.

--------------------------------------------------------------------------------
2. Functional Testing: Form Inputs (Single Item)
--------------------------------------------------------------------------------
[ ] Input text into "Project Title", "End-User or Implementing Unit", and "General Description". Ensure standard characters, numbers, and symbols are accepted.
[ ] Input text into "Mode of Procurement" and "Criteria for Bid Evaluation".
[ ] Select "Yes" and "No" alternately on the "Early Procurement Activity" radio buttons to ensure only one can be selected at a time.
[ ] Click on "Start of Procurement Activity" and "End of Procurement Activity". Verify that the flatpickr calendar dropdown appears. Select dates.
[ ] Input text into "Source of Fund", "Procurement Strategy or Tools", and "REMARKS".
[ ] Enter valid numerical values in "Estimated Budget/Approved Budget" (e.g., 5000, 10000.50).

--------------------------------------------------------------------------------
3. Functional Testing: Dynamic Form Elements & Calculations
--------------------------------------------------------------------------------
[ ] Click "Add Project" button. 
    - Expected: A new project item card (Item 2) should appear.
    - Expected: The new card should have empty fields.
    - Expected: The "Remove" (Trash icon) button should be visible on the new card (and potentially the first card depending on implementation).
[ ] Click "Add Project" multiple times. Verify that item numbers increment correctly (Item 3, Item 4...).
[ ] Input numbers into the "Estimated Budget/Approved Budget" field of multiple items.
    - Expected: The "Total Amount" at the bottom should automatically sum up the budgets in real-time.
[ ] Click the "Remove" (Trash icon) button on one of the items.
    - Expected: The item is removed from the form.
    - Expected: The "Total Amount" automatically recalculates and deducts the removed item's budget.
    - Expected: Item numbers adjust sequentially (if implemented).

--------------------------------------------------------------------------------
4. Form Submission: Save as Draft
--------------------------------------------------------------------------------
[ ] Fill out partial data in one or more items.
[ ] Click the "Save as Draft" button.
    - Expected: The system should save the current state without requiring all mandatory fields.
    - Expected: A success message should appear ("Draft saved successfully").

--------------------------------------------------------------------------------
5. Form Submission: Done (Submit)
--------------------------------------------------------------------------------
[ ] Fill out all required fields with valid data for one or more items.
[ ] Click the "Done" button.
    - Expected: Form submits successfully.
    - Expected: Redirects to the appropriate page (e.g., APP list or dashboard).
    - Expected: A success alert is displayed ("APP created successfully").

--------------------------------------------------------------------------------
6. Human Errors & Validation Testing (Error Handling)
--------------------------------------------------------------------------------
[ ] Submit Empty Form: Click "Done" without filling out any fields.
    - Expected: The page should block submission and display validation error messages for all required fields.
[ ] Invalid Budget Input: Attempt to enter letters or negative numbers (-500) into the "Estimated Budget/Approved Budget" field.
    - Expected: The field should not accept letters (due to type="number").
    - Expected: The field should not accept negative numbers (due to min="0"). If submitted, backend validation should catch it and show an error.
[ ] Date Order Error: Select an "End of Procurement Activity" date that is EARLIER than the "Start of Procurement Activity" date. Submit the form.
    - Expected: Backend validation should reject this and display an appropriate error message (e.g., "End date must be after start date").
[ ] Exceeding Allocated Budget (If applicable): Enter "Estimated Budget" amounts whose total exceeds the "Allocated Budget" displayed at the top. Submit the form.
    - Expected: System should ideally warn the user or block submission, displaying an error like "Total budget exceeds allocated budget."
[ ] Missing Radio Button: Leave the "To be covered by an Early Procurement Activity" radio buttons unselected and submit.
    - Expected: Validation error should prompt the user to select an option.
[ ] Removing All Items: Attempt to remove all project items including Item 1. 
    - Expected: The system should either prevent removing the very last item, or if removed, submission should fail with an error stating "At least one project item is required."

--------------------------------------------------------------------------------
7. Alert Dismissal
--------------------------------------------------------------------------------
[ ] Trigger a validation error so the red alert box appears.
[ ] Click the 'X' (close) button on the alert box.
    - Expected: The alert box should disappear smoothly.
[ ] Trigger a success message so the green alert box appears.
    - Expected: The 'X' button should close the alert box.
